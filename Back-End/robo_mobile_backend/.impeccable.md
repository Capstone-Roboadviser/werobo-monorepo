# Design Context

Applies to the two FastAPI-rendered admin pages:

- `mobile_backend/admin_web.py` — universe management (versions, instruments, asset-role config, price refresh)
- `mobile_backend/admin_comparison_web.py` — snapshot-based universe comparison board with charts

Internal-facing only. The Flutter app in `Front-End/robo_mobile/` is out of scope for this pass.

## Users

Internal quant/research team at WeRobo (Korean robo-advisor for retail investors). They sit at a desk, often all day, doing work that feels operationally heavy: adding/removing tickers, wiring asset-role rules, running price refreshes, comparing universe snapshots against each other on a chart board. Language is Korean. They read numerical tables fluently — information density is a feature, not a bug. Job to be done: operate the instrument, not browse a dashboard.

## Brand Personality

Three words: **precise, professional, unsentimental.**

Voice is that of a senior quant tool — confident in the density of what it's presenting, uninterested in hand-holding, quiet about itself. The emotional goal is that the user feels they're operating an instrument, not navigating a website. No marketing warmth, no "welcome back," no empty-state pep talk.

## Aesthetic Direction

**Modern professional trading tool**, in the lineage of **TradingView** and **Koyfin**. Chart-first, tight type, hover-driven detail, subtle semantic color. Both light and dark themes are first-class, with a user-controlled toggle. Each theme is tuned from scratch — the dark theme is not the light theme with colors inverted.

**References:** TradingView, Koyfin.

**Anti-references (strict — do not drift toward these):**

- **Generic shadcn / bootstrap admin.** No pastel-tinted cards with diffuse drop shadows and Inter. The current design sits here; this is what we are moving away from.
- **Retro CRT cosplay.** No scanlines, phosphor glow, or green-on-black decoration. "Bloomberg-like" refers to density and professionalism, not 1990s terminal aesthetics.
- **Crypto / neon / web3.** No cyan-on-dark, no purple-to-blue gradients, no glow effects on borders.
- **Korean banking legacy.** No heavy flat blue, no stacked dropdown menus, no density without craft.

**Theme:** Both light and dark with an explicit toggle. Light is for daytime office work; dark is for long chart-heavy sessions. Each palette is designed deliberately.

## Design Principles

1. **Density earns its keep.** We don't pad for padding's sake. Information per square inch matters — but every piece of information must be legible at a glance. Use tight leading on tables, generous space above section headings, varied rhythm throughout. Do not apply uniform padding as a default.

2. **Type does the work.** The interface is mostly labels, numeric values, and column headers. Pair a refined sans for UI chrome with a distinctive numeric or mono face for values. **Tabular numerals on every column of numbers** (`font-variant-numeric: tabular-nums`). Do not reach for Inter, IBM Plex, Space Grotesk, or anything else on the reflex-fonts list.

3. **Color is meaning, not decoration.** Semantic color only: greens for positive/active/ok, reds for negative/danger, ambers for pending/warning, a single brand accent for selection and primary action. Any color outside this palette must justify itself. Neutrals are tinted toward the brand hue for quiet cohesion. No gradient text. No decorative color on chart lines that do not encode a variable.

4. **One canonical accent, used sparingly.** A single distinctive accent — not a generic medium blue — does one job: point the eye at the current version, the active selection, the primary action. Every extra use dilutes it. The existing `#133c7a`/`#20A7DB` pair is acceptable as a starting point, but consider a more specific hue that commits to this product.

5. **Both themes designed, neither default.** Do not ship a light theme and "auto-dark" it. Build the dark theme as its own intentional palette — background tinted (never pure black), semantic greens/reds re-tuned for dark surfaces, chart gridlines and axes re-weighted for low-contrast backgrounds. A user should not be able to tell which theme was "the real one."

## Constraints

- **Korean-first copy.** All UI labels stay in Korean. Font choices must have Korean (hangul) glyph coverage, or pair a Latin display face with a Korean system/web font that matches it tonally.
- **FastAPI-rendered HTML strings.** No build step. All CSS inline in the Python files. No npm, no Tailwind CDN — keep the output self-contained.
- **Server-rendered, then hydrated by vanilla JS.** Do not rewrite the interaction layer in a framework. Polish what is there.
- **Scope: polish + opinionated redesign.** Permission to re-group sections, tighten the layout, rethink component patterns. No removed or added features, no new API endpoints.
